The cockpit —
decision-first, almost.

A review of the admin "The Issue" cockpit (KeywordStrategy, spine branch) — where the strategist curates the moves and sends the issue to the client. Like its client twin, it's already been through a serious pass: a five-beat spine above the fold, everything else collapsed. The residual issues are about where the primary action lives relative to where the work happens, and a collapsed drawer that has become a catch-all — the same pattern as the client page's "Under the hood."

What's already right

✓

A real spine. Setup-readiness → header → StanceBar → POV → backing moves is a legible top-to-bottom decision sequence, not a section pile.

✓

Decision-first. Everything that isn't the spine is collapsed by default, so a cold cockpit opens to the choice, not a wall.

✓

Honest staleness. The "POV may be out of date" nudge appears only when cut/edited moves diverge from the drafted POV — no false alarms.

The findings

AC-F1

The "Send issue" button is at the top; the staging happens at the bottom

Medium · control↔object

"Send issue" lives in the IssueHeader at the very top, gated on canSend={stagedCount > 0}. But the operator builds that staged set in the BackingMovesQueue near the bottom of the spine (StanceBar → POV → Add rec → then the queue). So the workflow is: scroll down, stage moves, scroll back up to send — and while staging, the button that consumes your work is off-screen. The control is separated from the object it acts on.

RecommendationKeep the header's Send for the "I'm done, fire it" case, but add a sticky send bar (or a Send affordance docked to the BackingMovesQueue) that appears once stagedCount > 0 — so the operator can stage and send without a round-trip scroll. The staged count + Send should be visible from where staging happens.

AC-F2

"Supporting detail" has become a one-toggle junk drawer

Medium · disclosure

Below the spine, a single collapsed <details> holds eight unrelated surfaces: admin leads readout, a keyword-targets deep-link, the content-work-order lens, the trust-ladder panel, orient, content gaps, cannibalization, and a strategy diff — plus a competitor link. It's all-or-nothing: to reach cannibalization you expand everything. This is the exact pattern as the client page's "Under the hood" (ISSUE-F-adjacent) — a recurring "drawer of everything."

RecommendationSplit the one drawer into 2–3 labeled disclosures grouped by job (capture, content signals, targets). Better: define a single Disclosure / collapsible-section pattern in the design system and use it both here and in the client "Under the hood" — same summary chrome, same behavior, sub-grouped. One pattern, two screens.

AC-F3

The staleness nudge hardcodes amber instead of using tokens

Low · token drift

The "POV may be out of date" banner uses raw Tailwind text-amber-400 / border-amber-500/30 / bg-amber-500/5 rather than the --amber token (and the system's soft-warning surface convention). It renders correctly today, but it won't follow a palette change or the light theme the way tokenized warnings do — and it's a second way of spelling "warning surface."

RecommendationUse the warning tokens (var(--amber) + the soft-amber background/border convention used by StatusBadge / InlineBanner). Pairs with the broader tokenize-hardcoded-hex work in the sprint (WH-F3 / the color sweep).

AC-F4

"Add a recommendation" floats mid-spine without a home

Low · placement

A right-aligned secondary "Add a recommendation" button sits on its own line between the Drafted POV editor and the BackingMovesQueue. It's a meaningful action (mint into the curation queue) but visually it's an orphan — not attached to the queue it feeds, just floating in the gap between two sections.

RecommendationDock "Add a recommendation" to the BackingMovesQueue header (it's the queue's own action — it adds a row to it), the same way a SectionCard hosts its action in the header slot. Removes the orphan line and puts the control on the object it modifies.

02

Read in context

The cockpit and its client twin are the two best-architected surfaces in the app — both already decision-first with disciplined disclosure. Their shared residual lesson is the same: put the action where the work is (AC-F1, AC-F4) and don't let "collapse the rest" become one infinite drawer (AC-F2, mirroring the client "Under the hood"). That last one is a design-system opportunity — a real Disclosure pattern both screens consume.
