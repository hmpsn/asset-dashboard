UX Review · admin content pipeline

Content Pipeline
has two maps of itself.

A review of ContentPipeline — where operators plan, brief, write, and publish. The flow framing is right (a clear Strategy → Briefs → Posts → Publish spine), and it deep-links cleanly. The trouble is two navigation systems describe the same journey differently, the tab bar is a hand-rolled copy of a primitive you already ship, and a stack of alert bands sits between the user and the work.

The findings

CP-F1

The stepper and the tab bar describe the same flow differently

High · mental model

Two navigations sit one above the other. The WorkflowStepper says Strategy → Briefs → Posts → Publish; the tab bar says Planner · Calendar · Briefs · Posts · Subscriptions. They only partly agree: "Strategy" leaves the page, "Publish" and "Subscriptions" are the same destination under two names, and Planner + Calendar exist only in the tabs. The operator can't tell which control is the real navigation.

Now — two disagreeing maps
stepperStrategy ↗→Briefs→Posts→Publish
tabsPlannerCalendarBriefsPostsSubscriptions
"Publish" = "Subscriptions"; Strategy leaves; Planner/Calendar unmapped
Fix — one map, aligned labels
stepperPlan→Brief→Write→Publish
tabsPlannerBriefsPostsPublishCalendar
stepper phases map 1:1 to tabs; Calendar is a view toggle, not a phase

RecommendationMake the stepper and tabs the same four phases with identical labels (rename "Subscriptions" → "Publish", drive the stepper's "Strategy" to the Planner tab instead of off-page). Treat Calendar as a view toggle on the Planner/Posts data, not a peer phase. One journey, described once — the stepper shows where you are, the tabs are how you move.

CP-F2

The tab bar is a hand-rolled copy of the TabBar primitive

Medium · system drift

The sub-tab bar is built inline from <Button variant="ghost"> with a manual border-b-2 + mint-underline active state — re-implementing the exact TabBar component the system ships and that WorkspaceHome already uses. Same pattern, second implementation: they'll drift in padding, hit-target, and keyboard behavior.

RecommendationReplace the inline bar with the TabBar primitive (it already supports icons + the mint active state and handles arrow-key nav + the 44px hit target). The Export control moves beside it as an action, not a fake tab.

CP-F3

Up to four alert bands stack before the work

Medium · density

Between the stepper and the tab content sit, in order: the health summary bar, the content-decay alert, the cannibalization warnings, and the AI-suggested briefs block. For a workspace with issues that's four full-width notices to scroll past before reaching Planner or Briefs — and three of them are amber/red, so the page reads as alarmed before the operator has done anything.

Now — four pre-content bands
Health summary · 12 briefs · 4 posts
3 pages losing traffic ↗
2 cannibalization warnings
5 AI-suggested briefs
— tab content starts here —
Fix — one roll-up, rest in context
Health summary · 12 briefs · 4 posts
3 alerts & 5 suggestions ▾
— tab content starts here —

RecommendationKeep the neutral health summary as the one persistent bar; roll decay + cannibalization + AI suggestions into a single collapsible "Alerts & suggestions (N)" strip, or move each into the tab it acts on (decay → Posts, suggestions → Briefs). The operator should reach the work in one screen, not after a wall of warnings.

CP-F4

Mint and amber on static counts in the health bar

Low · color law

The health summary renders brief/matrix counts in text-accent-brand (mint) and the posts count in text-accent-warning (amber). These are static data, not actions or warnings — the same mint-on-data drift flagged on the Command Center (CC-F5). Amber on a normal post count also implies a problem that isn't there.

RecommendationCounts are data → neutral bright text (or blue if you want them to read as metrics); reserve amber for the decay/cannibalization states that are genuinely warnings. Folds into the repo-wide color-law sweep already in the sprint.

CP-F5

The Export menu is hand-rolled — a missing system primitive

Low · system gap

The Export dropdown builds its own open state, click-outside handler, positioned panel, and per-row CSV/JSON buttons inline. It works, but it's a bespoke menu — and the design system has no Dropdown/Menu primitive, so the next menu (and there will be one) gets hand-rolled too, slightly differently.

RecommendationAdd a Menu / Dropdown primitive to the system (trigger + positioned panel + click-outside + keyboard dismiss baked in) and adopt it here. This is the one genuine new-component need surfaced so far — worth its own roadmap item.

The pipeline's instinct — frame content as a journey — is exactly right and worth protecting. The fixes are about saying that journey once (CP-F1), reusing the primitives you already built (CP-F2), and clearing the runway to the work (CP-F3). CP-F4 folds into the existing color sweep; CP-F5 surfaces the first real new-component need (a Menu primitive). All logged to the index and sprint.
