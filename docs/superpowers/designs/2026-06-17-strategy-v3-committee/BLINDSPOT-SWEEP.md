# Strategy v3 — Blind-Spot Sweep (synthesis)

**Date:** 2026-06-18  ·  **Method:** 6 adversarial lenses (grounded in repo) + synthesis, via Workflow `strategy-v3-blindspot-sweep` (w0mqrdtxe).
**Verdict:** all findings are additive completeness gaps inside the locked design (D1–D7) — none contradict the committee verdict.

## Executive read

The blind spots cluster into three shapes, and almost none destabilize the locked design — they are completeness gaps inside it. First and most important: the curation loop's two endpoints are both unbuilt and partly fictional in the design's 'reuse' framing. The outbound doorbell (no email/notification fires on a rec send — 'recommendations_ready' is trapped in a 14-day audit cooldown) and the inbound spine (Add·$ has no SKU and no rec→checkout path; Discuss has no thread substrate; client Approve has no write route or read filter) are all net-new, not reuse — confirmed against the code. Second, the committee's own Grafts 1-5 already locked the architectural answer to the recurring axis/concurrency/decline collisions that surface under five lenses, so those re-raisings collapse into a tight set of fold-into-plan implementation items: extend the carry-over to copy lifecycle fields, make the Graft-2 single-writer module transactional, convert the deny-list public projection to an allow-list, and route all 'active' readers through one isActiveRec predicate. The single biggest risk is the monetization spine being a dead button — the mockup's flagship $499 CTA, the mixed free+paid screen, and the above-cap projections all assume infrastructure (a strategy SKU, a per-tier included-allowance model, banded numbers) that doesn't exist; shipping the client overview before resolving this either ships a non-functional button or over-promises in a way the impactBand cap was explicitly built to prevent. None of this contradicts the locked D1-D7 or the committee verdict — it is all additive — with one true sequencing hazard: v3 P1 deletes components while v2's 'delete legacy layout' phase is still pending on the same flag, which can break the flag-OFF prod build unless v2's deletion runs first. The genuine Josh-decisions are narrow and mostly monetization-and-sequencing shaped; the deep data-model findings, though severe, have known correct defaults a competent planner applies given the grafts.

## The 6 decisions for Josh (each with recommended default)

### D1. For v1, do we ship Approve + Discuss only (gating Add·$ behind a child flag), or build the full rec-monetization spine — generic strategy_addon SKU + rec→cart bridge + real PRODUCT_MAP entries for keyword_gap/topic_cluster — before the client overview ships?

**Recommended default:** Ship Approve + Discuss for v1; render Add·$ only where rec.productType is already set (metadata/schema/accessibility/brief) and gate the net-new paid-topic path behind a child flag. The curation loop's value is Approve/Discuss; the $499-on-net-new-topics path is a separable monetization track and is currently a dead button (mapToProduct returns no product for those types).

### D2. Is the included-vs-paid boundary tier-driven (Premium = all included, Growth/Free = priced where assignedTo='client'), or do we build the mockup's per-rec mixed free+paid screen?

**Recommended default:** Tier-driven. Drive every CTA off assignedTo + productType, never a hardcoded 'Included' string, and snapshot the tier at send/approve time so a mid-trial Approve doesn't re-price at day 14. The mixed per-rec screen is a net-new monetization model — defer.

### D3. Does v3 P1 run AFTER the still-pending v2 'delete legacy layout' phase (hard prerequisite), or dark-launch behind a new child flag 'strategy-v3' while the v2 legacy OFF branch is still live on prod?

**Recommended default:** Make the v2 legacy deletion a hard prerequisite for v3 P1 — run it first to reach one layout, then build v3 on it. Nesting a child flag inside a still-two-branch flag invites a flag-OFF prod build break when v3 deletes a component the legacy branch imports. Grep-confirm every deletion is unreferenced before removing it.

### D4. How does the client learn a rec was sent — a new 'curated_recs_sent' email in the 3/day action bucket, or another channel?

**Recommended default:** A new 'curated_recs_sent' EmailEventType in the action bucket, batching one curation session into a single 'N recommendations ready' email deep-linking to the hub. Do NOT reuse 'recommendations_ready' — its 14-day audit cooldown silently swallows curated sends. Without a doorbell the 'hub they return to' premise has no trigger.

### D5. What happens when a SENT rec's underlying condition is genuinely fixed mid-discussion — and who is authorized to fire spend-bearing Approve/Add·$?

**Recommended default:** Exempt clientStatus in {sent,discussing,approved} from the destructive '✓ Auto-resolved'→completed sweep; transition fixed sent recs to a positive 'we handled this' state feeding WinsSurface, recording an outcome only with truthful attribution. Gate Add·$ and spend-spawning Approve to client_owner (new requireClientOwner guard); allow client_member to Discuss / request-intent. Both change endpoint behavior, so decide before speccing the routes.

### D6. Is lost_query a new Act-queue rec type (6-part registration incl. an OV-branch mapping) or a re-homed bespoke card?

**Recommended default:** Keep it a re-homed bespoke card for MVP. Making it a rec type requires an OpportunityValue branch it doesn't have (so it'd be unsortable and band-less) plus the rec→ActionType→ScoringConfig chain. Only promote it to a rec type if Act-queue curation of lost queries is a hard requirement, and then re-rate to L behind the Graft-2 policy registry.

Do NOT reuse 'recommendations_ready' (14-day cooldown swallows it). This is a Josh decision only insofar as the channel/copy/cadence is a product call; the plan can default to email-via-action-bucket.

### RN5. Auto-resolve mints a client-visible '✓ done' lie on a SENT rec the client is mid-discussion on — and there is no defined fixed-condition close semantics for sent recs  _(important)_
- **Missing:** The probe 'when the underlying condition is genuinely fixed on a SENT rec, what does the client see' is unaddressed. Graft 1's separate axis prevents the LEAK into the active list but does NOT add a clientStatus guard to the destructive auto-resolve sweep itself.
- **Grounding:** server/recommendations.ts:2391-2415 verified (force-sets completed + '✓ Auto-resolved' string for any non-completed/non-dismissed rec); server/routes/recommendations.ts records platform_executed outcome on completion; RECOMMENDATION_TRANSITIONS (state-machines.ts:100) verified single flat map. No clientStatus guard in the auto-resolve loop.
- **Recommendation:** Decide the sent-rec auto-close behavior: exempt recs with clientStatus in {sent,discussing,approved} from the destructive '✓ Auto-resolved'→completed sweep; when a sent rec's condition is genuinely fixed, transition it to a NEW positive terminal state the client reads as 'we handled this' (feeds the WinsSurface 'we called it' proof, not a cold ✓), and record an outcome only with attribution reflecting reality (platform_executed only if the agency did the work). ## Fold-into-plan (20) — planner applies the default; no Josh decision needed

**FP1. _(blocker)_ Merge carry-over must explicitly copy ALL lifecycle-axis fields (clientStatus/sentAt/throttledUntil) — 'extends verbatim' is FALSE**
  → In the lifecycle phase, add explicit carry-over of clientStatus/sentAt/throttledUntil (and struckAt/cascade metadata) onto every matched merged rec, via a dedicated carry-over branch that runs for EVERY matched oldRec regardless of RecStatus. Add a regen-preserves-lifecycle integration test (send a rec, regen, assert clientStatus still 'sent') as a phase exit gate alongside the committee's strike-never-completed test. This is mechanical once Graft 1's axis exists.

**FP2. _(blocker)_ Concurrency: whole-blob read-modify-write means curate/send/client-approve race against in-flight regen (last-write-wins) — Graft 2's single-writer module must enforce atomicity**
  → Make the Graft-2 single-writer module wrap every blob mutation in db.transaction() that re-reads the set INSIDE the txn (not the stale route-handler copy), applies the single-field delta, recomputes summary, upserts. Make generateRecommendations re-read existing and re-apply lifecycle-axis fields in the same synchronous tick as saveRecommendations so a curate/approve during synthesis survives. Route the new curate/send endpoints through the same per-workspace mutex the regen scheduler uses. Add a concurrency test. This is implementation of an already-locked Graft, not a new decision.

**FP3. _(blocker)_ Public projection is a deny-list (spread + strip-3-dollar-fields) — every new admin-only lifecycle field auto-leaks to the client and breaks the flag-OFF byte-identical gate**
  → Convert the public rec serialization to an explicit ALLOW-list of client-safe fields (or strip the new admin-only fields in the same commit they are introduced — make it a checklist item). Add BOTH the committee's flag-OFF byte-identical snapshot test AND a flag-ON test asserting no admin-only key (throttledUntil, struck*, supersession*, operator notes) appears in the public payload, on /api/public/recommendations/:ws, as a P2 exit gate before any client surface reads the field. Decide whether the client even needs clientStatus on the wire or only the server-side ==='sent' filter.

**FP4. _(blocker)_ Three additional 'active' readers count struck/throttled/sent recs as live opportunity — Graft 1 only names the two auto-resolve loops**
  → Define ONE shared predicate isActiveRec(rec) (status not completed/dismissed AND lifecycle not struck/throttled-until-future AND clientStatus not in {sent,approved,declined}) and route computeRecommendationSummary, the operational-slice counter, ActQueue, and the public GET projection through it. Enumerate every reader in the pre-plan audit (grep 'status !== ' + 'status === ' + 'pending'). Add a test asserting summary.topRecommendationId is never a struck/throttled/sent rec.

**FP5. _(blocker)_ recommendationSchema + public PATCH whitelist hardcode the status/type enums — new values are silently DROPPED on every reload unless added in lockstep**

**FP6. _(blocker)_ Client Approve has no write path + no read filter — public PATCH only accepts the 4 internal RecStatus values, public GET filters only ?status/?priority, and there's no second transition map**
  → Build a CLIENT_REC_TRANSITIONS map (system→sent→approved|declined|discussing) + a dedicated authenticated client route POST .../recommendations/:ws/:recId/respond {action} that mutates ONLY the lifecycle axis (never RecStatus, never the completion side-effects). Add the separate GET .../curated read (Graft 4) returning clientStatus='sent' recs with its OWN query key (queryKeys.client.curatedRecommendations) wired into both wsInvalidation branches for RECOMMENDATIONS_UPDATED. Keep the raw read path + shared key untouched so the byte-identical gate holds. Integration-test the real public read path.

**FP7. _(important)_ content_decay/cannibalization are BOTH Act-queue rec types AND deliverable adapters — no de-dup means the same page can be double-sent (rec + deliverable) with two approval flows**
  → Use the Graft-2 per-RecType policy registry to add a sendChannel: 'rec' | 'deliverable' field. For RecTypes with a registered deliverable adapter (content_decay, cannibalization), the Act-queue Send routes to the DELIVERABLE spine and the rec reads its lifecycle from client_actions state — NOT an independent clientStatus. Add a guard that a rec with an active deliverable cannot also be clientStatus='sent'. Test: sending a content_decay rec creates exactly one client-facing surface.

## Out-of-scope (5) — explicitly deferred

**OS1. Extracting recommendations to a per-row table** — The whole-blob storage model is the root cause of the concurrency and carry-over fragility, and a per-row table would dissolve several folded-in items at once. But it is a large migration (schema + rowToX + mapper + all readers/writers + public projection lockstep) that the committee explicitly scoped OUT of MVP — the Graft-2 single-writer + transaction wrapper is the chosen in-MVP mitigation. Record it as the strongest post-MVP architecture lever, not a v3 blocker.

**OS2. Per-tier 'included rec types' monetization model (the literal mixed free+paid client screen)** — The mockup's screen showing a free 'Approve refresh · Included' alongside a paid '$499 add-on' for the same client requires a net-new per-tier included-allowance model that does not exist (today included-vs-paid is all-or-nothing by tier). The faithful MVP mapping is tier-driven CTA labels. The mixed-screen model is a separate monetization track to defer unless Josh wants it as a first-class v3 goal.

**OS3. Full per-workspace reporting-timezone field** — The 'available X days' drift is fixed for MVP by shipping server-authoritative day-counts and UTC-floored throttle boundaries. A configurable per-workspace reporting timezone is only warranted if multi-region clients demand it — defer until there's demand.

**OS4. C's admin 'This month's client view' preview/narrative-control panel** — The committee already deferred this explicitly (dissent #2): it is net-new admin surface the audit never scoped, and the operator narrative-control 'wow' is the cheapest thing to defer. Carry as a P6/post-MVP nice-to-have; the lean base should not pay for it up front. The staleness/curation-health surface reuses StrategyStalenessNudges, not C's rebuild.

**OS5. Auto Stripe refunds in v1** — There is no refund route and the lifecycle is one-directional once charged. The MVP rule is manual-refund-as-a-task with clear 'contact your strategist' copy + a work-order-exists strike guard. Building a real automated refund flow is a separate billing track — defer, and do NOT promise auto-refunds in UI.
