## The 5 new pr-check rules

### Rule 1 — `feedback-module-reintroduction`

Prevents re-introduction of the `FeedbackWidget` component or `/api/feedback` routes that
were fully retired in PR 1.0a (migration 091 dropped the `feedback` table).

```ts
{
  name: 'feedback-module-reintroduction',
  pattern: '',
  fileGlobs: ['*.ts', '*.tsx'],
  severity: 'error',
  message:
    'FeedbackWidget and /api/feedback routes were retired in PR 1.0a (migration 091 dropped the table). ' +
    'Do not re-import FeedbackWidget or add feedback route handlers. ' +
    'Use the requests subsystem (/api/public/requests) for client submissions instead. ' +
    'Add // feedback-module-reintroduction-ok if intentional.',
  rationale:
    'The feedback table is gone. Any new route or import would fail at runtime silently if the ' +
    'table is never recreated.',
  claudeMdRef: '#code-conventions',
  excludeLines: ['feedback-module-reintroduction-ok'],
  customCheck: (files) => {
    const hits: CustomCheckMatch[] = [];
    const importRe = /import[^'"`]*['"`][^'"`]*FeedbackWidget[^'"`]*['"`]/;
    const routeRe = /router\s*\.\s*(get|post|put|patch|delete)\s*\(\s*['"`][/]api[/](public[/])?feedback/;
    for (const file of files) {
      if (!/\.(ts|tsx)$/.test(file)) continue;
      const content = readFileOrEmpty(file);
      if (!content) continue;
      if (!importRe.test(content) && !routeRe.test(content)) continue;
      const lines = content.split('\n');
      for (let i = 0; i < lines.length; i++) {
        const line = lines[i];
        if (!importRe.test(line) && !routeRe.test(line)) continue;
        if (/^\s*\/\//.test(line)) continue;
        if (hasHatch(lines, i, 'feedback-module-reintroduction-ok')) continue;
        hits.push({ file, line: i + 1, text: line.trim() });
      }
    }
    return hits;
  },
}
```

### Rule 2 — `keyword-strategy-action-type`

Prevents new code creating `client_actions` with `sourceType: 'keyword_strategy'`. The
type was deprecated in PR 1.0b (migration 092 archived all existing rows; the write path
was removed). Read paths are preserved with deprecation notices but no new rows should be
created.

```ts
{
  name: 'keyword-strategy-action-type',
  pattern: '',
  fileGlobs: ['*.ts', '*.tsx'],
  pathFilter: 'src/',
  severity: 'error',
  message:
    "'keyword_strategy' as a ClientActionSourceType was deprecated in PR 1.0b. " +
    "Do not create new client_actions with sourceType: 'keyword_strategy'. " +
    "Surface keyword data via the SEO Strategy page's existing approval workflow. " +
    "Add // keyword-strategy-action-type-ok if intentional (e.g. migration, type definition).",
  rationale:
    'Prevents accidental recreation of the deprecated keyword_strategy action creation path. ' +
    'The write endpoint was removed in PR 1.0b; a new call would hit a 404.',
  claudeMdRef: '#code-conventions',
  excludeLines: ['keyword-strategy-action-type-ok'],
  exclude: ['shared/types/', 'server/db/migrations/'],
  customCheck: (files) => {
    const hits: CustomCheckMatch[] = [];
    // Matches: sourceType: 'keyword_strategy' or type: 'keyword_strategy' in call sites
    const re = /(sourceType|type)\s*:\s*['"]keyword_strategy['"]/;
    for (const file of files) {
      if (!/\.(ts|tsx)$/.test(file)) continue;
      if (!file.includes('/src/')) continue;
      if (file.includes('/shared/types/') || file.includes('/migrations/')) continue;
      const content = readFileOrEmpty(file);
      if (!content || !re.test(content)) continue;
      const lines = content.split('\n');
      for (let i = 0; i < lines.length; i++) {
        const line = lines[i];
        if (!re.test(line)) continue;
        if (/^\s*\/\//.test(line)) continue;
        if (hasHatch(lines, i, 'keyword-strategy-action-type-ok')) continue;
        hits.push({ file, line: i + 1, text: line.trim() });
      }
    }
    return hits;
  },
}
```

### Rule 3 — `send-for-review-anti-pattern`

Prevents re-introduction of the old dual-button labels "Send for Review" and "Flag for
Client" in JSX. PR 1.4 consolidated all 13 admin send surfaces to a single
"Send to client" button with an optional inline note. The dual-button pattern is explicitly
retired.

```ts
{
  name: 'send-for-review-anti-pattern',
  pattern: 'Send for Review|Flag for Client',
  fileGlobs: ['*.tsx'],
  pathFilter: 'src/',
  severity: 'error',
  message:
    '"Send for Review" and "Flag for Client" are retired labels (PR 1.4). ' +
    'Use "Send to client" (single button) + optional inline note. ' +
    'See docs/workflows/ui-vocabulary.md §Admin Send Convention. ' +
    'Add // send-for-review-anti-pattern-ok if intentional.',
  rationale:
    'PR 1.4 unified all 13 admin send surfaces to a consistent single-button optional-note ' +
    'pattern. Re-introducing the dual-button pattern would create inconsistent UX.',
  claudeMdRef: '#code-conventions',
  excludeLines: ['send-for-review-anti-pattern-ok'],
}
```

### Rule 4 — `prediction-showcase-ungated`

`PredictionShowcaseCard` on `OverviewTab.tsx` must always appear behind `!winsEnabled`.
PR 1.3 established that when `client-wins-surface` is on, both surfaces must not be active
simultaneously. A bare `<PredictionShowcaseCard` without the guard breaks this invariant.

```ts
{
  name: 'prediction-showcase-ungated',
  pattern: '',
  fileGlobs: ['*.tsx'],
  pathFilter: 'src/',
  severity: 'error',
  message:
    'PredictionShowcaseCard must be gated with !winsEnabled (PR 1.3: client-wins-surface flag ' +
    'makes both surfaces mutually exclusive). Pattern: ' +
    '{!winsEnabled && clientIntel?.weCalledIt !== undefined && <PredictionShowcaseCard .../>}. ' +
    'Add // prediction-showcase-ungated-ok if intentional.',
  rationale:
    'When client-wins-surface flag is enabled, WinsSurface on the Insights tab replaces ' +
    'PredictionShowcaseCard on the Overview tab. Both active simultaneously is a product bug.',
  claudeMdRef: '#phase-per-pr',
  excludeLines: ['prediction-showcase-ungated-ok'],
  customCheck: (files) => {
    const hits: CustomCheckMatch[] = [];
    const cardRe = /<PredictionShowcaseCard/;
    const guardRe = /!winsEnabled/;
    for (const file of files) {
      if (!/\.tsx$/.test(file)) continue;
      if (!file.includes('/src/')) continue;
      const content = readFileOrEmpty(file);
      if (!content || !cardRe.test(content)) continue;
      const lines = content.split('\n');
      for (let i = 0; i < lines.length; i++) {
        const line = lines[i];
        if (!cardRe.test(line)) continue;
        if (/^\s*\/\//.test(line)) continue;
        if (hasHatch(lines, i, 'prediction-showcase-ungated-ok')) continue;
        // Check the same line and the two preceding lines for the guard
        const context = lines.slice(Math.max(0, i - 2), i + 1).join(' ');
        if (guardRe.test(context)) continue;
        hits.push({ file, line: i + 1, text: line.trim() });
      }
    }
    return hits;
  },
}
```

### Rule 5 — `inbox-action-queue-strip`

`ActionQueueStrip` must not appear in `InboxTab.tsx`. Per §5.6 of the parent spec, the
in-Inbox priority strip was removed in PR 1.2 (the Insights page ActionQueueStrip is
retained — it serves a different compass purpose). This prevents it from creeping back in.

```ts
{
  name: 'inbox-action-queue-strip',
  pattern: 'ActionQueueStrip',
  fileGlobs: ['InboxTab.tsx'],
  severity: 'error',
  message:
    'ActionQueueStrip must not appear in InboxTab.tsx (removed per §5.6 of client IA spec, PR 1.2). ' +
    'The Inbox carries urgency through chip counts and inline amber borders. ' +
    'ActionQueueStrip on the Insights page is fine — only InboxTab.tsx is restricted. ' +
    'Add // inbox-action-queue-strip-ok if intentional.',
  rationale:
    'The in-Inbox priority strip was compensating for the admin-shaped IA fragmented urgency. ' +
    'The new Inbox section headings carry that signal instead.',
  claudeMdRef: '#ui-ux-rules',
  excludeLines: ['inbox-action-queue-strip-ok'],
}
```

## New doc: `docs/rules/inbox-section-routing.md`

A reference doc covering the three-section routing rule introduced in PR 1.2. Every
developer touching InboxTab or adding a new client-action source type must read this.

**Content outline:**

```markdown
# Inbox Section Routing

## Three sections

| Section | Contains | Routing rule |
|---------|----------|-------------|
| Decisions | approval_batches + client_actions without a note | batch.note == null; action.clientNote == null |
| Reviews | content briefs, posts, copy pipeline items | static — no note-based routing |
| Conversations | approval_batches + client_actions WITH a note | batch.note != null; action.clientNote != null |

## Note-based routing invariant

When an admin sends an item "to client":
- **No note** → lands in Decisions (trust-first, one-click approve)
- **With note** → lands in Conversations (opens a discussion thread)

This is enforced at send-time in `server/routes/approvals.ts` and
`server/routes/client-actions.ts`. The InboxTab reads `note`/`clientNote` presence
to sort items into sections.

## Feature flag

The three-section layout is gated behind `new-inbox-ia` (default `false`).
When the flag is off, the pre-PR-1.2 four-section layout renders unchanged.

## Adding a new client_action source type

1. Add the value to `ClientActionSourceType` in `shared/types/client-actions.ts`
2. Decide routing: does this action type ever carry a note? If not → Decisions only.
   If yes → follows the note-based rule above.
3. Add a renderer case to `ClientActionDetailModal`
4. Update this doc

## Adding a new batch source

Same as above. Check whether the batch type can carry a `note` column and follow
the routing invariant.
```

```markdown
## Admin Send Convention (PR 1.4)

All admin "send to client" surfaces use a single button + optional inline note:

| Old (retired) | New |
```

Under **Code Conventions**, add two bullets:

```markdown
- **Admin send convention** — all admin "send to client" surfaces use a single "Send to client"
```
