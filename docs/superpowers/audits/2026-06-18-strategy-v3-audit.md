## Verdict on the two parallelism claims

### Stage 1 (Phase 0 ‖ Phase 1) — parallelOK: **False**

REFUTED as truly-parallel; CONFIRMED as a serialized two-PR gate. The proposed-shape claim 'file-disjoint except shared/types/feature-flags.ts' is GROUNDED-CORRECT on the file inventory — that is the ONE shared file (verified: P0 removes strategy-command-center from FEATURE_FLAGS line 59, FEATURE_FLAG_CATALOG line 282, and the 'Strategy' group keys array line 324; P1 adds the v3 child flag to the same three locations). BUT they cannot run concurrently because assertFeatureFlagGroupingConsistency() is an IMPORT-TIME invariant (verified import-time export in feature-flags.ts) — a half-applied merge throws at module load, breaking BOTH branches' typecheck/build. So Stage 1 is two SEQUENCED PRs sharing one gate: Phase 0 lands its flag removal + legacy-branch collapse first (NOT a route-removal — seo-strategy Page and strategy ClientTab both survive; tab-deep-link-wiring.test.ts is touchesReadOnly since it parses receivers dynamically and the surviving v2 branch still calls searchParams.get('tab')), then Phase 1 rebases. The only true parallelism inside Stage 1 is the INTERNAL lanes within each PR. OPEN DECISION the plan author must resolve before Stage 1: RETIRE strategy-command-center + fresh v3 flag (cleaner, matches keyword-hub migration-135 precedent) vs REUSE it as the v3 umbrella (spec §0 line 6 vs roadmap conflict). RECOMMEND RETIRE + new strategy-curation-cockpit flag — preserves the byte-identical-OFF guarantee.

### Stage 2 (admin Track B ‖ client Track C) — disjoint: **True**

**Shared-hot files:** `server/routes/recommendations.ts`, `server/state-machines.ts`, `src/lib/wsInvalidation.ts`, `src/lib/queryKeys.ts`, `server/activity-log.ts`, `src/lib/wsEvents.ts`, `server/ws-events.ts`, `src/hooks/useWsInvalidation.ts`, `shared/types/intelligence.ts`, `server/intelligence/client-signals-slice.ts`, `src/components/strategy/index.ts (Track B internal only)`

Stage-2 Track B (src/components/strategy/* + src/hooks/admin/* + admin recommendation routes) vs Track C (src/components/client/* + public recommendation routes + email/middleware server files) ARE file-disjoint on their CORE domains — GROUNDED-CONFIRMED. The shared-hot files above are all either (a) already-merged Phase-1 contracts that neither track re-edits, or (b) APPEND collisions resolved by merge order P2→P4 + pre-committed skeletons. The TWO mechanisms that make the tracks safely parallel: (1) Pre-commit in Phase 1's contract commit the FULL enum/event/state-machine surface — all four rec_* ActivityTypes + CLIENT_VISIBLE_TYPES classification, CLIENT_REC_TRANSITIONS, RECOMMENDATIONS_DISCUSSION_UPDATED event constant, ClientSignalsSlice.recResponses field, both queryKeys namespaces' shapes — so Stage-2 tracks only ADD route handlers/UI, never edit shared enums. With those two, the disjoint-tracks claim HOLDS.
