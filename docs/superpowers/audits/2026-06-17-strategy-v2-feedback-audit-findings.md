## ⚠️ THREE FORKS TO RESOLVE IN THE BRAINSTORM (they gate everything downstream)

### Fork A — Send substrate (Slice 1). *All 5 send-types inherit this choice.*
The send-to-client mechanism has two substrates: the **legacy `client_actions`** model (cheap, proven by cannibalization, dual-writes a deliverable — but invests further in the model the platform is migrating away from) vs the **unified `client_deliverable`/`sendToClient`** path (architecturally correct long-term, more net-new, needs a generic admin send route + threading the `unified-send-to-client-bespoke-route` pr-check hatch). There is **no generic admin send HTTP route today** — either path needs one. `coerceType` silently falls back to `seo_edit`, so a missed registration corrupts data without erroring. **Pick the substrate before any per-type send work.**

### Fork B — Persistence-across-regeneration overlay (Slice 3). *Gates the entire "managed sets" direction (#6, #9a).*
`siteKeywords`, `opportunities`, and `topicClusters` are **AI-regenerated on every strategy refresh** (`keyword-strategy-ai-synthesis.ts:1416`; `replaceAllTopicClusters` on persist). A naive add/remove + PATCH works **until the operator clicks Refresh — then their curation silently vanishes.** Per the audit: *"until that overlay exists, the managed-set feature is a trust-destroying mirage."* Needs a **respected-edits overlay** (kept-set + removed-set the synthesis pass honors) — the *same* "persist-across-regeneration" constraint already specced (but unbuilt) for the cannibalization keeper-override roadmap item. **Decide the overlay model before building managed sets.**

### Fork C — Layer-2 flip safety (Slice 6). *Gates the client-dashboard 3-layer model.*
Raw system recommendations currently **power the client InsightsEngine Action Plan + Health-tab upsell.** Naively hiding raw Layer-2 to show only curated Layer-3 **empties those surfaces.** Decide deliberately what the client sees (raw vs curated-only vs both) before flipping.

## ⭐ North-star constraints (Josh, brainstorm kickoff — apply to EVERYTHING)
1. **Admin AND client UI/UX, always together.** Never design an admin surface without deciding its client counterpart in the same breath. Both halves of every feature.
