# Product Surface Rationalization

Wave 5 item: `platform-reliability-product-surface-rationalization`

This rule doc defines how we classify platform capabilities so we reduce cognitive load without deleting valuable capability.

Source of truth: `scripts/product-surface-map.ts`

## Scope

This pass audits major capability surfaces, not all 400+ feature-audit rows.

It classifies major capabilities by:

- placement (`first-class-navigation`, `context-triggered`, `progressive-disclosure`, `internal-only`)
- role (`client-differentiator`, `agency-operations`, `support-system`, `platform-infrastructure`)
- lifecycle (`active`, `flagged-dark-launch`, `deprecated-but-present`, `legacy-alias`)
- recommendation (`promote`, `keep`, `hide-behind-progressive-disclosure`, `deprecate-after-redirect-window`)

## Why This Exists

Without an explicit taxonomy, the platform drifts toward additive-only navigation and hidden duplication:

- too many first-class nav items
- legacy aliases that never sunset
- internal operator tools competing with client-deliverable workflows
- feature-flagged surfaces that never graduate or retire

## Operating Contract

When a new major surface is added or an existing one materially changes:

1. Add or update an entry in `PRODUCT_SURFACE_MAP` (`scripts/product-surface-map.ts`).
2. Set a concrete recommendation (`promote`, `keep`, `hide-behind-progressive-disclosure`, `deprecate-after-redirect-window`).
3. Mark `requiresHumanVerification=true` for non-obvious cuts, deprecations, or nav demotions.
4. Add evidence pointers to real routes/components/docs (no speculative placeholders).
5. Run:
   - `npm run verify:product-surface`
   - `npm run verify:product-surface -- --markdown`
   - `npx vitest run tests/unit/product-surface-map.test.ts`

## Human Verification Gates

These decisions must be explicitly approved by a human reviewer before rollout:

1. Any recommendation that demotes a currently first-class nav item.
2. Any recommendation that deprecates a legacy alias or redirect path.
3. Any recommendation that changes client-facing discoverability for monetizable surfaces.

The current queue is generated by filtering `requiresHumanVerification=true`.

## Current Wave 5 Deliverable

Wave 5 item 1 is complete when all are true:

1. Every canonical bounded context has at least one mapped major capability.
2. The human verification queue is explicit and non-empty where required.
3. Coverage gaps from `findProductSurfaceCoverageGaps()` are zero.
4. This doc and `FEATURE_AUDIT.md` are updated in the same PR.

## Non-Goals For This Item

- Full 400+ feature taxonomy pass (tracked as a separate roadmap item).
- Removing routes/features in this PR.
- Changing UX copy or visual design during the audit pass.
