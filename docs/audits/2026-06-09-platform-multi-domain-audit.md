## Top confirmed findings (adversarially verified)

### 1. Explicit "no credits" strategy generation silently spends DataForSEO credits — 4-day-old billing regression
**High · bug · S effort · strategy-keywords · CONFIRMED**
`server/keyword-strategy-generation.ts:163-169` promotes an explicit `seoDataMode: 'none'` to `'quick'` whenever a provider is configured, because `server/routes/jobs.ts:375` collapses "param absent" and "explicit none" with `|| 'none'`. The admin UI offers 'none' with the copy *"No DataForSEO credits used"* (`src/components/KeywordStrategy.tsx:544`) — and the UI only sends 'none' when a provider exists, i.e. exactly when the override fires. The verifier confirmed credits are genuinely spent (universe discovery runs at quick depth) and that this became unconditional in commit `a5644282` (2026-06-05, flag retirement). **Fix:** keep `undefined` distinct from explicit `'none'` through the normalization chain; honor explicit 'none' with zero provider calls, and add a test asserting it.

### 2. Internal JWT "member" role is not workspace-scoped on most admin API routes
**High · security · M effort · security-scoping · CONFIRMED**
The global admin gate (`server/app.ts:234-239`) accepts any valid internal JWT with no workspace check; `requireWorkspaceAccessFromQuery` (`server/auth.ts:123`) has **zero callers**. A member JWT scoped to workspace A can read workspace B's full AI intelligence (`POST /api/admin-chat`, `server/routes/ai.ts:39`), activity (`/api/activity`), debug prompts (`/api/debug/prompt` — whose comment claiming the guard "cannot be used here" is wrong), and AI usage. The tenant-boundary audit script is structurally blind to query/body-derived `workspaceId` (it only scans `:workspaceId` paths). **Fix:** apply `requireWorkspaceAccessFromQuery/FromBody` to every admin route deriving workspaceId from query/body; add a pr-check rule for unguarded query/body workspaceId reads; fix the debug.ts comment.

### 3. Deleting an approval batch orphans its deliverable mirror — permanent ghost card + silent false approvals
**High · bug · M effort · data-flow · CONFIRMED**
`deleteApprovalBatchForClient` (`server/domains/inbox/approval-batch-admin-mutations.ts:82-125`) deletes the legacy batch but never touches the `client_deliverable` mirror created at send time; the deliverable store has no delete/cancel function. The orphan stays `awaiting_client` forever in the live client Inbox. Worse (verifier finding): responding to the orphan returns `{handled:true}` from the missing-batch path, so the client "approves" withdrawn content with **no team email, no activity, no source write** — and a unit test currently asserts the orphaning behavior. The schema-plan family already has the correct template (`cancelSchemaPlanDeliverable` + `DELIVERABLE_UPDATED`). **Fix:** add `cancelDeliverable` via `findBySourceRef`, call it from batch delete, broadcast `DELIVERABLE_UPDATED`; audit client-action and work-order deletion for the same orphan pattern.

### 4. Unified client Inbox is never invalidated when admin sends approval batches / client actions
**High · bug · M effort · data-flow · CONFIRMED**
The live client Inbox renders exclusively from the deliverables query, subscribing only to `DELIVERABLE_*` (+copy/post/work-order events) — but the approval-batch and client-action dual-write mirrors never broadcast `DELIVERABLE_SENT/UPDATED`; they fire only legacy `APPROVAL_UPDATE`/`CLIENT_ACTION_UPDATE`/`APPROVAL_APPLIED`, which map to legacy keys, not `client-unified-inbox`. A client with the Inbox open never sees new Decisions appear or decided cards leave until refocus. The broadcast-pairs meta-test can't catch it (the events *are* handled — just not for this scope). The schema-plan path in the same bounded context proves the correct pattern (`schema-plan-admin-mutations.ts:97-102`). Verifier correction: the briefing leg is moot (briefing mirrors are born `completed`/`notification` and never client-facing). **Fix:** broadcast `DELIVERABLE_SENT/UPDATED` at the approval-batch and client-action mirror seams; add the pairings to the meta-test.

### 5. Audit bridge and analytics recompute fight over the same `page_health` insight row
**High · bug · M effort · insights · CONFIRMED**
Bridge #12 (`server/reports.ts:230-256`) upserts `page_health` with **hardcoded zero metrics** (clicks/impressions/ctr…) + audit fields; the analytics recompute writes real GSC/GA4 metrics to the same unique key, and `ON CONFLICT` replaces `data`, `severity`, `impact_score`, and `bridge_source` wholesale. Every audit run overwrites real traffic data with zeros (which feed client InsightCards sorting and AI prompts), downgrades analytics-computed `critical` to `warning`, and every recompute NULLs `bridge_source` (stripping stale-cleanup immunity). `scheduled-audits.ts` and `webflow-seo-audit-bridges.ts` already migrated to the dedicated `audit_finding` type with read-before-write — `reports.ts` is the lone straggler. **Fix:** migrate reports.ts to `audit_finding` (delete its page_health writer), or merge via `getInsight` + `cloneInsightParams` preserving metrics and bridge_source.

### 6. Copy pipeline generation runs synchronously despite a `background-only` operation contract — and embeds a full discarded brief generation per call
**High · performance · M effort · content-generation · CONFIRMED**
`POST /api/copy/.../generate` awaits `generateCopyForEntry` in-request, though the registry declares `copy-generation` `executionMode: 'background-only'` (`server/ai-operation-registry.ts:187`) — **executionMode is enforced nowhere** (verifier: registry-only, no runtime check, no pr-check, no test). Layer 4.5 unconditionally runs `generateBrief()` — gpt-5.4, 7000 maxTokens, researchMode, plus up to **four** `buildContentGenerationContext` assemblies (verifier corrected the auditor's count of two) — before the Claude call. Single-section regenerate triggers the same full discarded brief. Realistic proxy-timeout territory on Render. **Fix:** route single-entry generation through `copy-batch-jobs` (batch-of-one works today); reuse `entry.briefId` instead of regenerating; skip brief enrichment on section regenerate. Consider enforcing `executionMode` at the `callAI` boundary.

### 7. Blueprint generation route synchronously chains 1 + N AI calls in one HTTP request
**High · performance · M effort · content-generation · CONFIRMED**
`POST /api/page-strategy/:workspaceId/generate` awaits one Claude call then a **sequential `generateBrief()` per page** (gpt-5.4 + researchMode each — the code's own comment admits "30–90+ seconds"). No `background-generation-ok` hatch; the pr-check rule's allowlist doesn't include the route file and only matches post-response patterns, not sync awaits. Verifier corrections: a proxy timeout doesn't strand partial state (the loop completes server-side) — the real hazards are deploy/restart mid-loop and **retry-duplication**: the client sees an error while AI spend continues, and a retry creates a duplicate blueprint plus a second full N-brief fan-out. The sibling copy path already uses the job platform. **Fix:** convert to a background job with per-entry progress, or return after the blueprint and run the brief loop as a follow-up job.

### 8. `REQUEST_TRANSITIONS` state machine is defined, claims to close a known bug, but is wired into nothing
**Medium (downgraded from high) · bug · S effort · decision-intelligence · CONFIRMED**
`server/state-machines.ts:238-245` defines the map with a comment saying "This map closes that gap" — zero production imports. `updateRequest` writes any merged status; the PATCH route emails the client on **every** status change including illegal `closed→new` reopens. The guard-coverage contract test omits the requests entity, and the pr-check "unguarded SET status" rule misses the statement shape. Downgraded because all mutation paths are admin-gated — harms are convention violation, spurious client emails, and an actively misleading comment. **Fix:** wire `validateTransition` into `updateRequest`, add the entity to `GUARD_SIGNALS`, surface `InvalidTransitionError` as 400. Do the same for `MATRIX_CELL_TRANSITIONS` (same pattern, also unwired — see Decision/intelligence below).

### 9. Four admin crawl/AI operations run as synchronous requests with only a spinner
**Medium (downgraded from high) · ux · L effort · admin-ux · CONFIRMED**
LinkChecker (unbounded page crawl), AEO site review (up to 25 pages × AI), llms.txt (per-page AI loop on cold cache), and internal-links analysis all bypass the job platform — the pr-check rule's route allowlist doesn't cover their files. Downgraded because results persist server-side and reload on mount; real harms are missing progress/cancellation, proxy-timeout false failures, and duplicate concurrent runs. **Fix:** migrate to `server/jobs.ts` + `BACKGROUND_JOB_TYPES` + TaskPanel; extend the pr-check allowlist to `webflow-analysis.ts`, `aeo-review.ts`, `llms-txt.ts`.

### Refuted: "Free-tier clients have no path to pending inbox deliverables"
The verifier confirmed the nav gating code exists but found **three tier-ungated discovery mechanisms** (Overview action-needed banner deep-linking to the inbox, ActionQueueStrip on the free-tier briefing branch, and the approval-ready email), so the approve/decline loop works. Residual: a low-severity polish item — add a persistent Inbox nav entry for free tier when actionable deliverables exist.

## Cross-cutting themes

### A. Unified-inbox migration debris — docs actively misleading agents
Independently flagged by three auditors. Eight client components have **zero live imports** (`ApprovalBatchCard`, `ApprovalsTab`, `DecisionDetailModal`, `ClientActionDetailModal`, `SchemaReviewModal`, `SchemaReviewTab`, `OrderStatus`, `WeCalledIt`, plus `inbox/InboxTabLayouts.tsx`, `inbox/useInboxTabShell.ts`), yet:

- Dual-write module headers claim the mirrors are "dark/flag-gated" (`unified-inbox-read.ts:22-24`, `client-action-dual-write.ts:25`) — they run unconditionally; no flag check exists. This stale claim is causally adjacent to confirmed findings #3 and #4.
- `InboxTab` declares 9 vestigial props that ClientDashboard still fetches and threads.
- `InsightCards.tsx` (568 lines) + `useClientRawInsights` are dead, kept alive only by their own tests.

**Fix as one cleanup PR:** delete the dead files, strip the props/threading, then update CLAUDE.md's inbox component list (live set: `DecisionCard`, `InlineApprovalCard`, `DeliverableDetailModal`, `ProjectedReviewModal`, `PriorityStrip`, `SubmitRequestChooserModal`), rewrite inbox-section-routing.md's enforcement table, and correct the dual-write headers — same commit, per the project's own doc-sync rule.

### B. Contracts that exist but are enforced nowhere (the audit's signature defect shape)
- `REQUEST_TRANSITIONS` / `MATRIX_CELL_TRANSITIONS` defined, never imported (confirmed #8).
- `executionMode: 'background-only'` in the operation registry — no runtime/pr-check/test enforcement (confirmed #6).
- `bridge-authoring.md` claims a pr-check rule (`bridgeSource missing on upsertInsight inside bridge`) that **does not exist**, claims per-bridge feature flags that are retired, and lists 5 registered bridge sources with no implementation.
- `getWorkspaceHealthScore` peeks the all-13-slices cache key that no production path warms (cron warms a 7-slice key) — outcome-measurement prioritization silently never runs (`server/workspace-intelligence.ts:151-157` vs `server/intelligence-crons.ts:38-40`).
- The tab deep-link contract test misses the template-literal `adminPath` sender form — the majority authoring form (receivers verified fine today; future breaks ship silently).

### C. Silent-empty-fallback JSON parsing in the brand engine
`generateCalibrationVariations`, `refineVariation`, `importBrandscript`, `completeBrandscript` all parse with `parseJsonFallback` and **persist empty results as success** (zero variations, empty brandscripts, no-op refinements). None are registered operations; none validate shape. Fix: register the four operations, Zod-validate, return 502/retry on failure — mirroring `parseStructuredAIOutput` in `content-posts-ai.ts`.

### D. Grounding leaks in client-facing content data
- Brief prompts demand `difficultyScore`/`trafficPotential` even with no provider metrics — the model invents them and clients see color-coded authoritative badges (`server/content-brief.ts:1391-1425` → `ContentTab.tsx:437-450`). Add a `metricsSource: 'provider' | 'ai_estimate'` provenance flag or suppress.
- The `internal_links` AI-fix literally instructs the model to "use a plausible internal link like /blog" when suggestions are empty — fabricated 404s can flow to Webflow auto-publish (`server/routes/content-posts.ts:715-728`).
- Checklist AI-fix misfires: `word_count_target` only expands (worsens over-budget posts); hallucination fixes always rewrite section 0 regardless of where the flagged claim lives.
- Markdown-styled error placeholders (`*[Introduction generation failed…]*`) stored in HTML fields render literal asterisks and can reach auto-publish.
