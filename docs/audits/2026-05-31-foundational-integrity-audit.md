The audit is brutally consistent on one theme: **the platform writes the right data but does not wire the write events to the surfaces that prove value or set priority.** The recommendation engine, the AI advisor context, the monthly digest, and the ROI dashboard are all driven by snapshots/caches that the action paths (approvals, work-order completion, schema publish, content publish, manual strategy edits) never refresh — and the one table designed to terminate the ROI loop (`roi_attributions`) has zero writers.

## Top 5 Highest-Impact Findings

1. **Applied SEO changes never drop the matching rec off the priority list (capability i + ii).** `generateRecommendations()` (`server/recommendations.ts:638`) has exactly four callers — none in the approvals, work-order, or page-state domain (`server/routes/approvals.ts:458-517`, `:308-359`; `server/routes/work-orders.ts:68-92`). A client can approve and apply the exact title/meta change a "Quick Win: …" rec demanded (`recordSeoChange('approval')` fires, page state set `live` at `approvals.ts:463`), yet that rec stays `status='pending'` in `recommendation_sets` indefinitely. The auto-resolve branch (`recommendations.ts:1373`) only completes a rec when its merge key disappears on a full regen, and approving never deletes the `quick_wins` row (`server/quick-wins.ts:90-117`; only writer is keyword-strategy persistence). The advisor keeps citing the stale pending rec as the top priority (`server/admin-chat-context.ts:843`). **Both the headline "one priority" surface and the action→outcome loop are broken on the surface that matters most.**

2. **Four disconnected "priority" stores with no unifying rank or join column (capability i).** `recommendation_sets` (`003-per-workspace.sql:126`), `client_business_priorities` (`021-client-participation.sql:5`, client intent), `workspaces.business_priorities` (`048-business-priorities.sql:7`, admin intent — migration's own comment flags it "Distinct from the client_business_priorities table"), and `client_actions.priority` (`083-client-actions.sql:12`) all model "priority" with no FK linking them. `generateRecommendations` reads NEITHER intent store into its ranking (`recommendations.ts:659-670` consumes only `.learnings`/`.backlinkProfile`; final sort `recommendations.ts:1413-1418` is tier+impactScore only). The platform's headline differentiator — ONE reconciled ranked priority per client — is not derivable from the schema; reconciliation, when it happens at all, happens only inside an LLM prompt.

3. **The ROI loop is physically severed at the ROI hop — `roi_attributions` has zero writers (capability ii).** The only table designed to link an action to a measured ROI delta (`040-insight-resolution-tracking.sql:8`) is permanently empty: its sole writers `recordOptimization`/`measureOutcome` (`server/roi-attribution.ts:55,83`) have zero callers anywhere (an unchecked TODO at `docs/rules/analytics-insights.md:281` admits the `brief_published → recordOptimization` wiring was never done). It is still read by `monthly-digest.ts:69` and `learnings-slice.ts:48-49`, both always empty. Compounding this, the live, populated outcome table `action_outcomes` (`041-outcome-tracking.sql:31-45`) has **no monetary column** — outcomes are measured in clicks/position only — and the client ROI dashboard (`server/roi.ts:139`) reads neither outcome table. Three disjoint value stores, no shared key, no path carrying a recommendation through to a dollar figure.

4. **Manual strategy edits never regenerate the served priority list (capability i).** The admin PageIntelligence PATCH `/api/webflow/keyword-strategy/:workspaceId` (`server/routes/keyword-strategy.ts:413-534`) rewrites `quick_wins`, `content_gaps`, `keyword_gaps`, `topic_clusters`, `cannibalization_issues`, and the `keyword_strategy` blob, then only invalidates the intelligence cache — it never calls `generateRecommendations()` or `queueKeywordStrategyPostUpdateFollowOns()` (`keyword-strategy.ts:7-52` import list confirms). Because `recommendation_sets` is a frozen snapshot served verbatim once a row exists (`recommendations.ts:260-275`, no staleness check; `routes/recommendations.ts:39-43` regenerates only when null), the client keeps seeing the OLD ranked priorities that contradict the edited strategy until the next audit. The frontend `STRATEGY_UPDATED` handler doesn't even invalidate `shared.recommendations` (`useWsInvalidation.ts:353-370`).

5. **Monthly digest can't prove applied work and reports "0 measurable improvements" (capability ii).** The digest's `issuesAddressed` is filtered strictly on `resolutionStatus==='resolved'` (`server/monthly-digest.ts:78`), but neither approval-apply nor work-order completion resolves an insight to `'resolved'` — the action bridge only sets `'in_progress'` (`server/outcome-tracking.ts:147-166`), and only a completed `ClientAction` (a separate inbox surface) ever yields `'resolved'`. Separately, `roiHighlights = getROIHighlights()` (`monthly-digest.ts:69`) is hardwired to the dead `roi_attributions` table, so it is always `[]` and the AI summary prompt unconditionally emits `${roi.length} measurable improvements` = 0 (`monthly-digest.ts:244`). A month of real, approved-and-applied SEO fixes shows `pagesOptimized=0` from those surfaces on the one client-facing proof-of-value narrative.

## Coverage & Method

- **What was mapped:** the four state-change domains (approvals/work-orders, keyword-strategy edits, schema publish, content publish) against the three consuming surfaces that embody the platform's intended capabilities — the `recommendation_sets` single-priority list (i), the rec→action→outcome→ROI loop across `tracked_actions`/`action_outcomes`/`roi_attributions`/`roi_snapshots`/monthly-digest/ROI-dashboard (ii), and the cached `WorkspaceIntelligence` slices feeding the AI advisor + MCP tools (iii). Both wiring (Track A) and data-model/schema (Track B) were audited across ~466 server `.ts` files and 106 migrations.
- **Adversarial testing:** 11 analyzers produced findings; every finding was re-opened by a skeptic who actively hunted for closing wiring — server-side event subscribers, crons, DB triggers (only `067`/`068`, unrelated discovery size-guards, exist), reconciling reads, frontend invalidation, and alternate populated paths.
- **Outcome:** **39 findings confirmed**, **16 refuted/dropped** (skeptic found a closing path or the impact evaporated), **4 left unclear** (one verifier errored transiently; three hinge on live feature-flag state or an internal-only surface that can't be resolved from read-only source). The skeptic corrected impact downward on many cache-invalidation findings (≤5-min self-healing LRU staleness on internal/admin surfaces, not client-facing) and upward/held on the priority-list and ROI-loop structural gaps.
- **Deduplication:** the dead `roi_attributions` table is the single root cause behind 4 separate confirmed findings (digest roiHighlights, learnings-slice ROI, B-outcome-roi-loop, B-cross-cutting-structure) — merged into one Section-B entry. The schema-publish cache-invalidation gap was reported by 3 analyzers across siteHealth/pageProfile/contentPipeline — merged into one. `markSchemaApplied` and the four-priority-store fragmentation each appeared twice — merged.
- **Could not verify directly:** production data — local `data/dashboard.db` is empty (0 bytes); the 2 live clients run on the Render deployment, so live feature-flag overrides and actual `recommendation_sets`/`action_outcomes` contents were out of reach. Conclusions hold structurally regardless of flag state, but the four Unclear items specifically require checking the live `feature_flag_overrides` table.
