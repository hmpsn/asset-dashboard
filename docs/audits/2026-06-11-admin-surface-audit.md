# Admin Surface Audit — Strategy, Keyword Hub, Local SEO, Schema, Content Generation

**Date:** 2026-06-11 · **Base:** `origin/staging` @ `3fb7a181` · **Method:** 18-agent parallel audit — per surface, one UI/UX auditor + one functionality auditor, then a per-surface adversarial verifier that attempted to refute every "missing capability" and quick-win claim against the whole repo. Three additional agents swept admin-wide UI/UX consistency, cross-surface integration/consolidation, and the end-to-end admin journey. **Verification outcome: 187 claims checked — 164 confirmed, 23 adjusted, 0 refuted.** All findings below survive verification; adjusted claims are stated in corrected form.

---

## Executive summary

**Overall: the backends are dramatically better than the edges.** Every surface has a disciplined core — job-platform generation, transactional persistence, broadcasts, bounded AI calls — and every surface leaks value at the seams: swallowed errors rendering as false empty states, write-only data nothing reads back, finished server pipelines with no UI reaching them, and "next step" handoffs that drop their payload.

**The five recurring patterns across all surfaces:**

1. **Silent failure is the #1 UX defect class.** Strategy fetch errors render as "No keyword strategy yet" (inviting a credit-burning regeneration); the Local SEO panel returns `null` on error across all three mounts; the post editor shows "Saved" after a failed save. Same fix shape everywhere: stop `.catch(() => null)`, branch on `isError`, render `ErrorState` with retry.
2. **Write-only loops.** Outcome tracking records keyword baselines nothing reads back; persisted AI post-reviews are never re-read; Local SEO visibility snapshots accumulate as an unread time series; schema single-page generations aren't persisted at all. The platform is collecting the data to close its feedback loops — it just never reads it.
3. **Broken handoffs between surfaces.** AI Suggested → Brief drops the keyword; Page Intelligence → Schema sends `pageSlug` where the receiver wants `pageId`; WorkspaceHome's "client requests" CTA lands on the wrong tab. The integration *intent* is everywhere; the two-halves contract keeps failing on the receiving half.
4. **Finished pipelines gated off.** CMS-field schema publishing is complete server-side but unreachable behind a blanket `cms-` gate; the suggested-briefs subsystem has zero UI; calendar intelligence is dead code.
5. **Stale-data gaps from missing WS wiring.** Production KeywordCommandCenter has zero `useWorkspaceEvents` handlers against a server that broadcasts on every mutation.

**Headline consolidation recommendation (§7):** finish the Keyword Hub P5 cutover and retire KeywordCommandCenter + RankTracker + `seo-ranks` — but *only after* porting the Local SEO visibility panel into the Hub, since KCC is currently its only keyword-mode mount. Keep Strategy separate ("plan" vs "operate") and connect the two with per-entity deep links, three of which are currently broken.

**Top 10 quick wins across the platform** (all S-effort, all verified not-already-implemented):

|---|---------|-----|

## 1. Strategy (`seo-strategy`)

### Verdict

The Strategy surface has the strongest core engineering in the admin app and the weakest edges. The generation pipeline is genuinely excellent — background-job based with deterministic progress and pre-mount job recovery (`src/components/KeywordStrategy.tsx:81-111, 206-226, 746-750`), TOCTOU-safe usage metering with refunds on every failure path (`server/keyword-strategy-generation.ts:155-165, 386-402`), a `BEGIN IMMEDIATE` transactional persist with history snapshots, activity logging, and idempotent outcome-tracking actions (`server/keyword-strategy-persistence.ts:130-277`), and a closed-candidate-set hallucination guard in AI synthesis (`server/keyword-strategy-ai-synthesis.ts:644-689`). But the polish hides real problems: every read-path error is swallowed into a false "No keyword strategy yet" empty state, a failed keyword-add offers a "Try Again" that launches a credit-burning full regeneration, two subcomponents sit outside React Query and show stale data after regeneration, and the `strategyUx` payload silently lost all site-keyword metric evidence when persistence moved to table-as-truth. The surface is also analysis-rich and action-poor — Quick Wins, Low-Hanging Fruit, topic-cluster gaps, and Intelligence Signals are all read-only dead ends. Adversarial verification scored the underlying audits 30 confirmed / 6 adjusted / 0 refuted, so the findings below are solid.

### What's working

- **Generation is fully on the background-job platform** — `BACKGROUND_JOB_TYPES.KEYWORD_STRATEGY` with progress streaming, 409 dedup at both the job layer and an in-process `activeGenerations` guard, and recovery for jobs started before mount (`server/routes/jobs.ts:353-430`, `KeywordStrategy.tsx:206-226`). Exactly the contract `docs/rules/background-generation.md` prescribes.
- **Race-safe usage metering** — `incrementIfAllowed` reserves the slot *before* async AI work, and every failure/no-op/sanitizer-only path refunds it (`server/keyword-strategy-generation.ts:153-165, 386-403, 619`).
- **Exemplary persist path** — transaction, history snapshot, activity log, per-keyword outcome actions with net-new diffing, `STRATEGY_UPDATED` broadcast, and intelligence-cache invalidation in one place (`server/keyword-strategy-persistence.ts:130-277`); single read assembler collapses five divergent read paths (`server/keyword-strategy-assembler.ts`).
- **RefreshOrderingPrompt is excellent decision UX** — intercepts Generate when local SEO data is stale and offers full-refresh / generate-anyway / cancel with reason-specific copy (`src/components/keyword-strategy/RefreshOrderingPrompt.tsx:15-22`), backed by a bidirectional local-SEO↔strategy staleness contract (`server/local-strategy-sync.ts`).
- **Real cost guards and an honored no-spend contract** — per-workspace monthly provider-call ceiling, branded+declined hard filters (`server/keyword-strategy-universe.ts:57-75`); `seoDataMode: 'none'` makes zero provider calls and unrecognized modes degrade to none rather than auto-spending (`server/keyword-strategy-generation.ts:92-105`). Credit/time estimates shown per settings choice (`KeywordStrategy.tsx:616-620, 698-703`).
- **The client feedback loop is closed** — declined keywords hard-filter the pool, requested keywords get priority plus one-click Add to Strategy, and a nudge fires when feedback is newer than the last generation (`KeywordStrategy.tsx:147-161, 326-332, 449-462`). Content Gaps rows carry real Draft Brief / Generate Brief deep links with `fixContext` state (`src/components/strategy/ContentGaps.tsx:75-94`).

### Issues

#### 🔴 Critical

None. The core pipeline (generation, persistence, metering, broadcast) is sound; every finding below is an edge-path, consistency, or actionability problem.

#### 🟠 Major

- **Failed "Add to Strategy" renders as "Strategy Generation Failed" whose Try Again launches a full, credit-consuming generation job.** `addRequestedKeywordMutation.onError` writes into the *same* `error` state used for generation failures (`src/components/KeywordStrategy.tsx:157-160`), which renders `<ErrorState title="Strategy Generation Failed">` with a Try Again that calls `generateStrategy('full')` (`KeywordStrategy.tsx:752-758`) — a multi-minute job that burns DataForSEO credits. Spot-checked: real, severity fair. Keyword-add failures need their own inline/toast surface.

- **Strategy fetch errors are swallowed and render as "No keyword strategy yet."** `useKeywordStrategy`'s queryFn does `.catch(() => null)` (`src/hooks/admin/useKeywordStrategy.ts:44`), so any 500/network failure resolves to `data=null` and the empty-state card (`KeywordStrategy.tsx:777-788`) — telling an admin whose workspace *has* a strategy that it vanished, with a misleading prompt to regenerate. (Verified: worse than reported — the catch is per-attempt, so the configured `retry: 2` at `:48` never fires and `isError` can never be true.) The aux query has the same pattern (`:56-57`).

- **StrategyDiff and BacklinkProfile hand-roll `useState`+`useEffect` fetches — the "What Changed" panel shows a stale diff after regeneration.** Both violate the mandatory React Query convention (`src/components/strategy/StrategyDiff.tsx:13-24`, `src/components/strategy/BacklinkProfile.tsx:40-53`). When regeneration completes in-session, `STRATEGY_UPDATED` invalidates 16 React Query keys (`src/lib/wsInvalidation.ts:38-57`) but can't touch local state — the components stay mounted, so the diff panel keeps showing the *previous* run until a full remount. Both also swallow errors silently, and BacklinkProfile re-issues its DataForSEO-backed fetch on every tab switch with no cache.

- **`strategyUx` reads `siteKeywordMetrics` from the stripped workspace blob — site-keyword explanations silently lost all volume/difficulty evidence.** `buildKeywordStrategyUxPayload` builds `siteMetricByKeyword` from `options.strategy?.siteKeywordMetrics` (`server/keyword-strategy-ux.ts:458`), but persistence force-strips that field from the blob — the table is the sole store (`server/keyword-strategy-persistence.ts:108-113`). The GET route passes the raw blob even though it holds `assembled.siteKeywordMetrics` from the assembler at `server/routes/keyword-strategy.ts:261`. Result: every site-keyword explanation ships with `metric=undefined` — no "X monthly searches", no `valueReasons` — on admin GET, diff, and the client portal payload, for every workspace generated since the strip. Verified confirmed; this is the textbook silent-data-loss pattern the read-before-write rule targets.

- **`POST /api/seo/competitors` mutates workspace data with no `broadcastToWorkspace` and no `addActivity`** (`server/routes/seo-provider.ts:181-194`) — violating mandatory Data Flow Rules 1 and 4. The Strategy tab caches `competitorDomains` for 10 minutes (`src/hooks/admin/useKeywordStrategy.ts:52-70`), so competitor changes from another tab/session — or the silent auto-discovery write inside generation (`server/keyword-strategy-seo-data.ts:163, 238-242`) — leave the settings panel and CompetitiveIntel stale until `staleTime` expiry.

- **Entity-resolution intelligence slice reads `ws.keywordStrategy.pageMap`, which never exists — page-level entity candidates are dead code.** `server/intelligence/entity-resolution-slice.ts:97-109` gates page-scoped candidates on a blob field that persistence strips before storage (`server/keyword-strategy-persistence.ts:99-105`) and the assembler documents as table-only (`server/keyword-strategy-assembler.ts:51`). The branch is always false, so schema entity grounding never receives page-level keyword candidates. Should read `listPageKeywords()` like `seo-context-slice.ts:69-85` does.

- **Four Laws violation: orange as the active/selected state on the SEO Data Mode picker and Auto-Discover button.** Selected cell gets `border-orange-500/50 bg-orange-500/10 text-accent-orange` (`KeywordStrategy.tsx:609-613`, icons `:601/:637`, Auto-Discover `:662-666`) while the sibling Page Limit picker correctly uses teal (`:691-695`) — internally inconsistent within one panel. Verified: `BRAND_DESIGN_LANGUAGE.md:68` reserves orange for changes-requested/urgent, the per-component map lists KeywordStrategy as teal, and there is no documented DataForSEO-branding exception (zero "DataForSEO" hits in the design doc).

- **Forbidden hue family: `sky-*` in KeywordStrategyGuide** (`src/components/strategy/KeywordStrategyGuide.tsx:46-48, 164, 169`) — should be `blue-*` per Law 2. (Verified: not a one-off — `sky-` appears in 8 places across `src/components/`, mostly guide components, and pr-check mechanizes violet/indigo/rose/pink but not sky. Fix should sweep all guides or land the pr-check rule first.)

- **Collapsibles and segmented pickers lack accessible state; selection is color-only.** No collapsible on the surface passes `aria-expanded` — Settings header (`KeywordStrategy.tsx:554`), Business Context toggle (`:716` — verified: that one is a `Button`, not ClickableRow), StrategyDiff (`StrategyDiff.tsx:38`), CompetitiveIntel rows (`CompetitiveIntel.tsx:214, 271`). The SEO Data Mode and Page Limit pickers are de-facto radio groups with no `role=radiogroup`/`aria-checked` and color-only selection (`KeywordStrategy.tsx:604-623, 686-706`), failing WCAG 1.4.1. Bonus bug: the Ranking Distribution "Not ranking" segment uses `bg-[var(--surface-3)]` — identical to the track background — so that share of the bar is invisible (`:852` vs `:857`; legend dot at `:864` has the same problem). (Verified: `ClickableRow` already accepts `aria-expanded` via its `ButtonHTMLAttributes` passthrough at `src/components/ui/ClickableRow.tsx:5-6`, so this is missing *usage*, not missing primitive support.)

- **Keyword-gap data appears twice on the same page under contradictory names.** `strategy.keywordGaps` renders as the always-expanded "Raw Competitor Evidence" card (`src/components/strategy/KeywordGaps.tsx:21-44`, mounted `KeywordStrategy.tsx:897`) and again inside CompetitiveIntel's collapsible "Keyword Gaps" section, which falls back to the *same* `cachedKeywordGaps` rows when live data is empty (`KeywordStrategy.tsx:926`, `CompetitiveIntel.tsx:148-149, 268-297`). In the common no-live-data case, identical rows appear twice — one framed "evidence only", one framed "opportunities" — with no cross-reference.

- **The Guide documents a "create a brief from a Quick Win" capability that does not exist.** `KeywordStrategyGuide.tsx:83` and `:190` promise it; `QuickWins.tsx:16-47` and `LowHangingFruit.tsx:23-50` render zero buttons or links. ContentGaps got the Draft/Generate Brief pattern (`ContentGaps.tsx:75-94`); the two improvement-focused sections never did. In-app docs actively mislead admins into hunting for a button that isn't there.

## 4. Schema (`seo-schema`)

### Verdict

Schema is the deepest surface in the platform and architecturally the strongest: a site-wide AI plan feeds a deterministic lean generator with a bounded AI budget (120 calls/run, `server/schema-suggester.ts:35-37`), Wikidata entity resolution stays inside the intelligence-slice boundary, generation runs on the background job platform, and publishing has a real server-side validation gate, version history, retraction, outcome tracking, and a GSC impact panel. But it is undermined by wiring holes at exactly the seams admins hit most: single-page generation output is never persisted server-side (it silently reverts on reload), the entire CMS-field delivery pipeline is unreachable from the UI, client schema approvals dead-end with no apply path, published state lives only in session memory, and bulk publish fires a full Webflow site publish per page from a browser loop. The verification pass confirmed 37 of 39 claims and all 8 spot-checked majors — these are real, not audit noise. Closing the persistence and CMS-publish holes would convert this from an impressive generator into a closed-loop schema operations system.

### What's working

- **Full lifecycle in one surface** — plan → generate → validate → publish → version history/rollback → retract → client review → outcome tracking → impact measurement, all covered by `server/routes/webflow-schema.ts`.
- **Background job platform used correctly for generation** — bulk scan via `startJob(BACKGROUND_JOB_TYPES.SCHEMA_GENERATOR)` with progress + cancel (`useSchemaSuggesterGeneration.ts:137-155`), plan generation with 409 + jobId conflict guards against concurrent regeneration (`server/domains/schema/schema-plan-admin-mutations.ts:35-44`).
- **Real pre-publish validation gate** — structural + Google rich-results rules return 422 before any Webflow write, and whole-site graph errors block bulk publish with matching UI affordance (`webflow-schema.ts:400-431`, `SchemaSuggester.tsx:113`).
- **Deterministic lean generator with a well-designed role-override precedence chain** — UI page-type > active site plan > CMS collection mapping > saved type, with `roleDecisionDiagnostics` explaining each decision (`server/schema-suggester.ts:205-302`); entity resolution respects the intelligence-slice boundary (`server/schema-suggester.ts:125-151`).
- **Broadcast → invalidation wiring is better than most surfaces** — `SCHEMA_PLAN_UPDATED`/`SCHEMA_SNAPSHOT_UPDATED`/`SCHEMA_CMS_MAPPING_UPDATED` all have registry handlers covering admin and client keys (`src/lib/wsInvalidation.ts:84-112, 267-274`), and publish follow-ons connect to llms.txt regen, recommendation regen, and intelligence cache invalidation (`webflow-schema.ts:481-482, 569-581`).
- **SchemaCompletenessWidget is a model "next action" pattern** — aggregates validation findings by field, ARIA-correct progressbar, deep-links each missing field to the exact settings tab + focus anchor (`SchemaCompletenessWidget.tsx:96-136`); the impact panel closes the loop with honest 28-day before/after GSC comparisons (`SchemaImpactPanel.tsx:90-117`).

### Issues

#### 🔴 Critical

- **Single-page generation/regeneration results are never persisted — output silently reverts.** `POST /api/webflow/schema-suggestions/:siteId/page` (`server/routes/webflow-schema.ts:378-392`) returns the generated schema but never writes it to the snapshot and never broadcasts; the frontend only updates local React state (`useSchemaSuggesterGeneration.ts:173-234`). "Add Page" and per-page "Regenerate" output vanishes on reload, and worse, any `SCHEMA_SNAPSHOT_UPDATED` refetch (a publish on another page, another admin's partial save) clobbers unsaved generations mid-session via the snapshot effect at `useSchemaSuggesterGeneration.ts:45-66`. Verified: `updatePageSchemaInSnapshot` only updates existing pages (`schema-store.ts:110-111`), so the fix needs insert-if-missing.

- **CMS-field schema publishing is unreachable from the UI despite a complete server pipeline.** `publishSchemaToCmsField` (`webflow-schema.ts:106-200`) writes sanitized JSON-LD into mapped collection fields, the mapping UI exists, and the generator computes `cmsDeliveryStatus: 'ready'` per item (`schema-suggester.ts:439-468`) — but the Publish button is hidden for every CMS page (`!page.pageId.startsWith('cms-')` at `SchemaPageCard.tsx:308`) and bulk publish filters them out (`useSchemaSuggesterPublishingWorkflow.ts:214, 277`). An admin can map a field, generate 200 CMS schemas, see "CMS field ready" in diagnostics — and has no path to publish any of them except copy-paste per item. (Verified: partially adjusted — `cmsDeliveryStatus` is on the server type and serialized in the admin view, but the frontend `SchemaPageSuggestion` type at `src/components/schema/schemaSuggesterTypes.ts:19-33` omits it, so the fix needs the type added first; and `publishSchemaToCmsField` resolves collections from the snapshot, so the persistence fix above is a prerequisite for single-page-generated CMS items.)

#### 🟠 Major

- **Published state is session-only — reload reverts every published page to "unpublished."** `published` initializes as an empty Set (`useSchemaSuggesterPublishingWorkflow.ts:26`) and nothing seeds it from `page.lastPublishedAt`, even though the server annotates publish dates on every snapshot response (`webflow-schema.ts:216-221`, `server/serializers/client-safe.ts:185`). After reload: the "Published" badge disappears and the Publish CTA reappears (`SchemaPageCard.tsx:308-352`), Retract is hidden (`SchemaPageCard.tsx:375`), and "Publish All (N)" over-counts and re-publishes live pages (`useSchemaSuggesterPublishingWorkflow.ts:277`). Spot-checked real; only mitigation is an unchanged-hash check on the CMS path (`webflow-schema.ts:145`) — static pages get full rewrites + site publishes.

- **Client schema approval dead-ends: approved batches can never be applied.** "Send to Client" creates items with `field:'schema'` (`useSchemaSuggesterPublishingWorkflow.ts:55-77`), but the apply gate `isClientApplyableFields` (`shared/applyability.ts:55-59`) only allows `seoTitle`/`seoDescription`, so `applyApprovedBatchItems` rejects the whole batch (`server/domains/inbox/approval-batch-apply.ts:69-79`). No admin "apply approved" action exists, and `sentPages`/`sentToClient` are session-local state. Client approval produces no deployment — the review loop is decorative.

- **Bulk publish runs as a client-side loop firing a full Webflow site publish per page, with no failure summary.** `publishAllToWebflow` (`useSchemaSuggesterPublishingWorkflow.ts:211-225`) loops sequential browser POSTs, each with `publishAfter:true` (line 90) triggering `publishSite` per request (`webflow-schema.ts:493-499`). For the 100+ page sites the guide anticipates (`SchemaWorkflowGuide.tsx:51`), Webflow rate-limits will fail most publishes silently (response still reports `success:true` with `sitePublished:false`); closing the tab abandons the remainder; per-page failures land only inside collapsed cards (`SchemaPageCard.tsx:391-393`), so partial failure looks like success. Violates the background-generation rule — the generation path already uses the job platform, making the publish loop the inconsistent outlier.

- **Stale manual JSON edits silently override regenerated schema.** `getEffectiveSchema` prefers `editedSchemaJson[pageId]` (`useSchemaSuggesterPublishingWorkflow.ts:48-53`), but `regeneratePage` (`useSchemaSuggesterGeneration.ts:212-234`) and `runScan` never clear it; `onPageGenerated` only clears `manualDelivery` (`SchemaSuggester.tsx:106-109`). Post-regenerate, the preview, Publish, Copy, and both send-to-client paths all consume the old edit with zero indication outside edit mode (`SchemaEditor.tsx:31-36`) — regeneration appears to do nothing, and stale schema can ship to client sites. Survives full re-scans too, since pageIds are stable Webflow ids.

- **A single-page generation failure replaces the entire results view with a full-screen error.** `generateSinglePage`'s catch sets `scanError` (`useSchemaSuggesterGeneration.ts:194-196`) and `SchemaSuggester.tsx:291-303` early-returns a full ErrorState whenever it's set, even with full results intact underneath. ## 5. Content Generation (`seo-briefs` / `content` / `content-pipeline` / `calendar`)

### Verdict

This is the deepest, best-engineered surface in the platform — and also the one with the most expensive-looking buttons that do nothing. The core path (keyword → SERP-scraped, GSC/GA4/provider-enriched AI brief → multi-step post generation job → provenance-gated review → unified Webflow publish) is genuinely strong, with high server hygiene: state-machine status transitions, `parseJsonSafe` everywhere, schema-validated AI outputs, version snapshots with coalescing, and a centralized WS-invalidation registry. The rot is at the edges, in two clusters. First, **write-only / half-wired features**: the persisted AI review is never read back (`StoredAIReview` ships to the DB and dies there), the entire suggested-briefs accept/dismiss/snooze subsystem has zero UI, `suggestPublishDates` is dead code, matrix bulk actions are silent no-ops, and the flagship "AI Suggested → Create Brief" button discards the keyword it was clicked for. Second, **silent failure**: the post editor shows "Saved" after a failed save, and most ContentManager/PostEditor mutations swallow errors into `console.error`. Verification was unusually clean — 33/38 claims confirmed, 5 adjusted, 0 refuted — so everything below is load-bearing.

### What's working

- **Background-job platform used correctly for the two expensive flows.** Brief and post generation run as jobs with progress, 409 single-flight guards, cancellation, and TaskPanel tracking (`server/content-brief-generation-job.ts:379-418`, `server/content-posts.ts:226-298`); ContentBriefs handles 409 with a clear toast and converges via broadcasts on remount (`src/components/ContentBriefs.tsx:126-185`).
- **Brief generation is exceptionally well-grounded** — GSC related queries, provider keyword metrics, GA4 page performance, scraped references + live SERP/PAA, page-profile analysis, EEAT recommendations, decay context, strategy cards, and outcome learnings, all via the sanctioned `buildContentGenerationContext` builders (`server/content-brief.ts:1065-1374`, `server/content-brief-generation-job.ts:113-235`).
- **Provenance contract faithfully enforced.** `factual_accuracy` / `no_hallucinations` are forced `pass=false` + human-review server-side with a claim-evidence ledger (`server/routes/content-posts.ts:81-101`, `server/content-review-evidence-ledger.ts:104-131`), plus a defense-in-depth client guard (`ReviewChecklist.tsx:93-104`).
- **Publish path properly unified** — manual and auto-publish-on-approval both call the same `publishPostToWebflow()`; auto-publish is an idempotent `CONTENT_PUBLISH` job that short-circuits on `webflowItemId` and surfaces failures as durable activity (`server/routes/content-posts.ts:362-388`, `server/content-publish-job.ts:38-93`).
- **Broadcast + invalidation discipline is complete**: every brief mutation broadcasts and logs (`server/routes/content-briefs.ts:90,136,191,216,268-270,301`), and `src/lib/wsInvalidation.ts:159-173` centrally maps content events to admin + client query keys.
- **Color/token hygiene and hard UI contracts hold**: zero purple/violet/indigo/rose across 15+ audited files, z-index exclusively via tokens, the `?tab=` deep-link two-halves contract respected (`ContentPipeline.tsx:50-64`), and the RichTextEditor focus-guard + autosave flush race handling matches `docs/rules/rich-text-content.md`.

## 6. Admin-wide UI/UX

### Verdict

The admin shell is fundamentally sound: a coherent 6-group, color-semantic sidebar with persisted collapse state (`src/components/layout/Sidebar.tsx:58-152`), a genuinely good ⌘K command palette with fuzzy match and recents (`src/components/CommandPalette.tsx`), per-tab `ErrorBoundary` + lazy chunks (`src/App.tsx:540-544`), centralized WS→React Query invalidation (`src/hooks/useWsInvalidation.ts`), and clean class-based forbidden-color greps — zero violet/indigo/rose/pink classes, zero `text-green-400`, zero purple in `src/components/client/` — proof the pr-check layer works for what it can see. The two systemic failures are exactly where automation can't see: **navigation metadata is triplicated and has already drifted** across Sidebar, CommandPalette, and Breadcrumbs (missing tabs, stale tabs, disagreeing `needsSite` gating, three names for one screen), and **the StatCard `iconColor` hex prop is an open loophole in the Four Laws** — violet `#a78bfa` and green-as-success `#4ade80`/`#22c55e` ship today on the most-viewed admin screens. Around the edges, the shell breaks its own conventions: a palette action labeled "Run Audit" that doesn't run an audit, a badge count fetched outside React Query that goes stale, and a dead `TaskPanel` component that tests and CLAUDE.md still treat as production.

## 7. Integration map & consolidation

### Current wiring — what already connects

The platform's cross-surface plumbing is more extensive than typical, and the server side is genuinely unified rather than cosmetically so:

- **One keyword merge point on the server.** `buildKeywordCommandCenterModel` (`server/keyword-command-center.ts:1511-1614`) merges strategy site keywords/metrics, `page_keywords`, content gaps, keyword gaps, tracked keywords, GSC latest ranks, lost-visibility rows, client feedback, and local-SEO candidates into one row model. Strategy and KCC routes share `assembleStoredKeywordStrategy` and table-resolved `site_keyword_metrics` — volume/difficulty agree across surfaces by construction.
- **The `fixContext` router-state contract is well-engineered.** `App.tsx` owns the state (`src/App.tsx:216-248`); every receiver guards on `targetRoute` to reject stale contexts (`ContentBriefs.tsx:87-105`, `ContentPipeline.tsx:98-102`, `PageIntelligence.tsx:113-126`). Page Intelligence → Briefs carries rich payloads — `optimizationIssues`, `recommendations`, `contentGaps`, `autoGenerate` — that feed brief generation (`PageIntelligence.tsx:172-190`, `ContentBriefs.tsx:426-436`).
- **Server bridges close real loops.** Content decay → suggested briefs (`server/routes/content-decay.ts:38-65`) surfaces in the pipeline's AiSuggested panel; strategy generation reconciles rank tracking and queues llms.txt + recommendation regeneration (`server/keyword-strategy-follow-ons.ts:32-66`); `local-strategy-sync.ts` computes bidirectional staleness that Strategy renders as a refresh-ordering prompt and reverse-staleness nudge (`KeywordStrategy.tsx:419-437, 804-836`).
- **Centralized cross-invalidation.** `STRATEGY_UPDATED` / `RANK_TRACKING_UPDATED` / `LOCAL_SEO_UPDATED` each invalidate keywordStrategy + keywordCommandCenter + rankTracking + localSeo query keys (`src/lib/wsInvalidation.ts:40-73, 402-418`) — the keyword surfaces stay mutually fresh.
- **Local SEO is an embedded capability, not a silo.** One panel, three modes (strategy/keywords/page), mounted inside Strategy, KCC, and Page Intelligence (`LocalSeoVisibilityPanel.tsx:11, 261-370`), with keyword-level local posture annotating KCC rows and Page Intelligence keyword chips.
- **Schema is keyword-aware through the slice architecture.** `server/schema-intelligence.ts:75-105` pulls `seoContext.pageKeywords`, siteInventory, entityResolution, eeatAssets via `buildWorkspaceIntelligence` rather than ad-hoc reads.
- **The fold-in pattern is proven.** Content Pipeline already consolidated Planner/Calendar/Briefs/Posts/Subscriptions into one tabbed surface with a workflow stepper linking back to Strategy (`ContentPipeline.tsx:29-35, 115`), and the legacy `calendar` Page redirects into it (`src/App.tsx:451`).

### The keyword triangle

**It's a quadrilateral today, and the consolidation is already mid-flight.** With the `keyword-hub` flag OFF in production (the default — `shared/types/feature-flags.ts:37`), an admin sees the same keyword on four surfaces:

| Surface | Route | Data identity | Rank/clicks shown |
|---|---|---|---|
| **Keyword Strategy** | `seo-strategy` | Planning snapshot: `page_keywords` / `site_keyword_metrics` / workspace JSON blob | Frozen at `generatedAt` (`KeywordStrategy.tsx:299-307, 840-843`) |
| **Keyword Command Center** | `seo-keywords` (flag OFF) | Live operational merge of 10 sources (`server/keyword-command-center.ts:1511-1614`) | Live GSC overlay (`:1301-1311`) |
| **Keyword Hub** | `seo-keywords` (flag ON) | Same KCC model; consolidates Rank Tracker | Live, with disclosed 28-day window (`KeywordHub.tsx:467-471`) |
| **Rank Tracker** | `seo-ranks` | `tracked_keywords` + rank snapshots | Redirects into Hub when flag ON (`src/App.tsx:459`) |

The Hub is Wave 4 (`data/roadmap.json` `keyword-hub-wave4`, in_progress), with the sidebar already hiding Rank Tracker when the flag is ON (`Sidebar.tsx:68`). ## 8. The admin journey

### Verdict

The golden path is substantially built and better-integrated than most platforms this size: every lifecycle stage — onboarding → audit → strategy → keyword hub → local SEO → briefs → posts → schema → send-to-client → outcomes — has a real surface; cross-tool handoffs run through the deliberate `fixContext` mechanism; long-running work sits on a proper background-job platform with WS broadcasts and remount recovery (`server/jobs.ts:210-236, 273`; `SeoAudit.tsx:288-318`, `KeywordStrategy.tsx:206-226` — a page refresh mid-audit doesn't lose the run); and the send-to-client spine (`server/domains/inbox/send-to-client.ts` + `server/routes/deliverables.ts`) is genuinely well-engineered — state-machine-guarded transitions, Zod drift guards on both client and admin reads, staleness tracking, per-item Remind buttons. WorkspaceHome earns roughly a B+ as mission control, with a prioritized P1/P2/P3 Needs Attention queue (`WorkspaceHome.tsx:168-220`), churn signals, decay, anomaly alerts, an admin recommendation queue, weekly accomplishments, and a Meeting Brief tab. The chain breaks in three concentrated places: **progress visibility** (background jobs are invisible outside the tab that started them, and the audit run is a static spinner), **deep-link senders violating the project's own `?tab=` two-halves contract** (P1 actions landing on the wrong sub-tab, home metrics navigating to an orphaned tab), and **onboarding step 1 dead-ending** on two read-only pages that both point back at a dropdown the new admin may never have opened.

### Where the chain breaks — the five worst

**1. Background-job progress has no global surface — a 10–15 minute job goes dark the moment you navigate away.**
- The journey audit's framing: `TaskPanel.tsx` is a complete floating job panel (per-job label, message, percent from `job.progress/job.total`, cancel via `isBackgroundJobCancellable`) with zero importers, while `docs/rules/background-generation.md` and CLAUDE.md both designate it the global progress surface. App.tsx mounts Sidebar, Breadcrumbs, StatusBar (`App.tsx:491-548`) — no job panel.
- Consequence: a keyword-strategy run advertised at "~10-15 min" for 1000 pages (`KeywordStrategy.tsx:699-703`), an audit crawl, a local-SEO refresh, or a brief generation is visible only inside the originating tab. Navigate anywhere else and there is zero indication work is running, finished, or failed.
- *Reconciliation with §6:* the UI/UX sweep found PR #831 moved the job surface into NotificationBell (`NotificationBell.tsx:6,95` consume `useBackgroundTasks`), so the remedy is not "mount TaskPanel" but "delete TaskPanel, fix `CLAUDE.md:274`, and verify the bell actually delivers cross-tab running-job visibility + cancel." The underlying complaint — and the audit-spinner fix below — stands either way. The job state itself survives in `BackgroundTaskProvider` + WS; this is a surfacing problem, not a platform problem.

**2. The P1 "N new client requests" action lands on the wrong sub-tab.**
- `WorkspaceHome.tsx:171` pushes the highest-priority action with `tab:'requests'` and no queryString; the click handler navigates with no `?tab=` (`WorkspaceHome.tsx:549`).
- The requests page defaults its sub-tab to `deliverables` and only re-reads `?tab=` when `urlWorkspaceId` changes (`App.tsx:259-272` — effect deps `[urlWorkspaceId]`). Navigating from Home within the same workspace never changes that, so the admin lands on Client Deliverables while the requests they clicked live under the Requests sub-tab (RequestManager).
- Same sender-half failure: "content plan pages need review" (`WorkspaceHome.tsx:212`) targets content-pipeline with no `?tab=planner`, landing on the default Briefs tab (`ContentPipeline.tsx:50-55`) while review cells live in Planner.
- The platform's own two-halves deep-link contract, violated by its own mission-control surface.

**3. Onboarding step 1 ("Link Webflow site") dead-ends on read-only status pages.**
- The OnboardingChecklist step navigates to workspace-settings (`WorkspaceHome.tsx:231-238`), whose Connections tab renders Webflow as status-only with helper text "Linked via workspace dropdown" (`settings/ConnectionsTab.tsx:80-86`) — no link action.
- The other discovery path is worse: any `needsSite` tab shows "Go to Settings" → global `/settings` (`App.tsx:423-433`), where SettingsPanel also only lists linked/unlinked workspaces with "Link sites from the workspace dropdown" (`SettingsPanel.tsx:195-216`).
- The only place a site can actually be linked (paste token → fetch sites → pick) is inside the WorkspaceSelector dropdown (`WorkspaceSelector.tsx:54-75, 235-250`). A new admin is bounced between two settings pages that both point at a dropdown they may never have opened — at the very first step of the journey.

**4. The sidebar disables Requests for workspaces without a Webflow site — cutting off all client communication in exactly the early-journey state.**
- `Sidebar.tsx:98` marks `{ id:'requests', needsSite: true }` and the disable logic (`Sidebar.tsx:230`) greys it out when `!selected.webflowSiteId`.
- App.tsx's own gate disagrees: `requests` is *not* in `SEO_TABS` (`App.tsx:403-404`), so the route renders fine without a site — client requests, signals, client actions, and the deliverables pane have no Webflow dependency.

- **The admin Traffic Value card links into the client portal.** `clientPath(workspaceId,'roi')` (`WorkspaceHome.tsx:464`) — a full context switch out of the admin shell (sidebar, breadcrumbs, admin auth context) with no path back; if the HMAC token isn't honored by the client-portal auth path in that browser session, it lands on a client login.
